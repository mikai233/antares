# RPC Protocol Registry

`rpc-protocol.json` is the source of truth for internal protobuf RPC message ids.
It is generated by the local Gradle task `generateRpcProtocolRegistry`.

Why this file exists:

- internal RPC message ids need to stay centrally managed
- ids should not be scattered across `.proto` messages as per-message options
- client/gateway routing and internal RPC registration are separate concerns

Registry model:

- internal RPC protobuf messages live under `server-proto/src/main/proto/rpc/`
- shard entity-id extraction uses proto options such as `option (asteria.rpc.entity_id) = "..."`
- protobuf RPC message ids are registered centrally in `rpc-protocol.json`
- Asteria codegen consumes:
    - the proto descriptor set
    - this JSON registry
      and generates `GeneratedRpcProtocol`

What happens when adding a new internal RPC message:

1. define the message in `server-proto/src/main/proto/rpc/`
2. if the message goes through shard routing, add `entity_id` option in the proto message
3. run `./gradlew :server-proto:generateRpcProtocolRegistry`
4. review and commit the updated `rpc-protocol.json`

Client-facing messages that participate in internal remoting are discovered from:

- `client-proto/src/main/proto/client/msg_cs.proto`
- `client-proto/src/main/proto/client/msg_sc.proto`

## Framework Follow-up

The JSON registry is a project-side bridge for centralized RPC id allocation. The framework should eventually own this
flow so applications do not maintain `rpc-protocol.json` directly.

Target behavior:

- keep RPC ids centrally allocated
- generate or synchronize the registry from descriptor scanning
- avoid putting message ids directly into individual proto messages
- generate protobuf RPC registration from the framework-owned registry
